# 劫寨战斗 · 后续开发规划

> 编写日期：2026-07-23
> 状态基线：Demo 迭代 1–5 全落地、21 个 raid 单元测试绿、B3 平衡工具 `tools/raid_sim.js` 已交付、L1/L2 难度曲线双落带。
> 规划依据：`combat-mode-design.md` §8.1 三阶段蓝图、四份 GDD、`review-comprehensive.md`(P0/P1/P2 + S1–S4)、`demo-plan.md` §5.3 待办与"难度设计教训"。
> 目标：把"已做出好玩 demo"推进为"可上线、可扩展、可平衡的完整玩法"，并按优先级给出可执行的下一步。

---

## 0. 规划原则

1. **对齐官方蓝图 §8.1**：后续仍按「修循环 → 治背板 → 长尾留存」三阶段推进，但代码层已超额完成"修循环"，故下一步重心转向**难度闭环延伸 + 文档对齐 + 表现层补全**。
2. **复用已验证方法论**：本工程已沉淀一套可复用的难度标定方法论（见 `demo-plan.md` §5.3「难度设计教训」），所有关卡/武将扩张都须用 `raid_sim.js` 验证落带，禁止拍脑袋调数。
3. **严守架构红线**：逻辑/表现分离、`data.js` 为唯一数据源、所有随机经 `RollSource` 注入。新增系统不得污染 `src/core/` 与 SRPG。
4. **文档与代码同步**：规划把"代码已做 / 文档滞后"单列为一类，要求每阶段把对应 GDD/设计文档补齐，避免进一步漂移（R6 认知过载的根因正是口径不一）。

## 2. 阶段路线图

> 每阶段都标「范围 / 关键任务 / 验收 / 依赖」，阶段间按价值×风险排序，阶段内任务可并行。

### 阶段 A · 难度与内容闭环（最高优先级，复用 B3 工具）
**为什么先做**：B3 工具已就绪，难度方法论已沉淀，这是把"1 关好玩"扩展为"5 关可玩+阵容可选"的最低成本路径，直接兑现 S1 与 §6 难度曲线蓝图。

| 任务 | 范围 | 验收 |
|---|---|---|
| **A1 L3–L5 接入 raid_sim** | 在 `data.js` 按 §6 蓝图建 L3 大名府/L4 高唐州/L5 蔡京府；复用「难度设计教训」（坦克小兵墙+中等 BOSS、塔避大门、忌堆塔）；`--curve` 校验落带 | L3 55–70% / L4 45–65% / L5 40–60% 落带（逐关递减）；每关 50 场换种子稳健 |
| **A2 L4 反 combo 改博弈** | 法术塔驱散按 S2：周期脉冲 8s/1s、范围 3 格、不清召唤物、可骗出 | 吴用/鲁智深/时迁 在 L4 不被全废；单一最优解占比<60%（raid_sim combo 频率） |
| **A3 武将池扩张 8→12–15** | 新增武松/杨志/柴进/阮小七/张清等高辨识度将（数据+基准技能）；用 raid_sim 验证编队张力（N1：12 选 8≈495 组合） | 可行编队组合 ≥200；兵符/槽位张力有趣（5–8 个典型编队示例入 GDD-01 §③） |
| **A4 天气平衡** | 纠雪天偏易（移速惩罚/潜行暴露强度重标）；天气×关卡交叉验证 | 4 天气胜率极差<20%（消除"雪天必赢"） |
| **A5 浏览器端关卡接入** | `raid.html` 加关卡选择 + L2/L3 入口；复用 `LEVELS`/`getLevel` | 浏览器可玩 L1–L2（L3+ 视 A1 进度），零控制台报错 |

**依赖**：无。A1–A4 可并行（各关独立数据），A5 最后接入。

### 阶段 B · 表现层补全（玩家体验闭环）
**为什么第二**：代码循环已成立，但"新玩家能否自己玩懂""静音是否还'燃'"决定可上线性。B4 音效成本最低、收益最高（前次建议已标注"音频最快"）。

| 任务 | 范围 | 验收 |
|---|---|---|
| **B1 音效系统**（G3/B4/D5/P1-6） | WebAudio 合成关键事件音：部署/破墙/塔火/技能/绝技/星级/胜负；清单入各 GDD §⑥ | 静音→有声；高光时刻音画同步；零资源依赖 |
| **B2 教学引导**（G4/P1-7） | L1 五步脚本（部署→技能→撤兵→劫掠→拨济），首次出现动词时触发引导层 | 新玩家无外部帮助完成一局 |
| **B3 劫掠引导强化**（B2） | 粮仓高亮+靠近提示；拨济结算收益感（声望/自留对比） | 多数玩家主动劫掠而非直推核心 |
| **B4 UI 完善**（R6/HUD） | HUD 三层分层落实；星级"或"化；拨济/声望反馈强化 | 认知负荷下降（实测不打断节奏） |

**依赖**：无硬依赖，可紧接 A 阶段或并行。

### 阶段 C · 系统深度（组合空间）
**为什么第三**：武魂/关系/可互动物是 §8.1 阶段一/二的核心差异点（R4 灵魂差异化），但数值叠加有失控风险（R8），须用 raid_sim 持续覆盖。

| 任务 | 范围 | 验收 |
|---|---|---|
| **C1 武魂系统**（阶段一） | 每将 1 被动+1 小技；用 raid_sim 防 R8 失控 | 8 将武魂接入后胜率仍在目标带；无数值爆炸 |
| **C2 关系羁绊**（阶段二） | 高阈值羁绊（如鲁智深+林冲"步战双雄"），平衡覆盖 | 羁绊增益占比合理，不构成唯一解 |
| **C3 庄丁 6 兵种分化**（§4.6/迭代5待后续） | 基础庄丁→枪兵/弓手/盾兵等分化，难度按 A 方法论重标 | 兵种分化提升 combo 多样性，落带不变 |
| **C4 可互动物**（S3） | 火药桶（远程引爆）/烽火台（报警增强）/滚石（高地推落） | 引入 1–2 个即显著改变战术，涌现性上升 |

**依赖**：A 阶段（raid_sim 稳定）；C1–C4 顺序递进。

### 阶段 D · Meta 与长尾留存
**为什么最后**：依赖前序系统稳定，且部分需服务器（UGC）。

| 任务 | 范围 | 验收 |
|---|---|---|
| **D1 Meta 存档**（G5/P1-8） | `MetaState` 结构（声望/解锁/关卡进度）+ IO 在 `src/ui/`，`src/core/` 不碰 localStorage | 跨局进度持久化；刷新不丢 |
| **D2 养成回流** | 养成近似数值→实战 CSV（参考 `tools/sim.js` 思路）；养成影响实战 | 满级 vs 新号难度曲线分离合理 |
| **D3 招降 / 三幕脚本 / 拨济叙事**（R9/阶段三） | 招降耗声望+拨济+10%，进武魂池；关卡三幕（破门→巷战→斩首） | 招降不破坏 roster 平衡；三幕触发清晰 |
| **D4 每日/周常**（路线图） | 依赖服务器，本期可只做本地周常 | 长尾留存指标正向 |

**依赖**：B/C 阶段系统稳定。

### 贯穿线 · 文档对齐与平衡 CI（不单独成阶段，随各阶段完成）
- 每阶段把 1.2 列出的"代码已做/文档滞后"项回写对应 GDD（M1–M8/G1/N2/H1/H3/M7）。
- `raid_sim.js` 接入轻量 CI：每次 `data.js` 变更自动跑 L1–L5 难度曲线，输出落带告警，防止数值回归。
